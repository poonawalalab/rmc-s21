Instructions

Each paper review must:

  1. Describe the broad area the work fits into, and the specific problem being solved

  2. Explain the core idea of the solution

  3. Describe shortcomings, and mention any papers that overcame them

Submission

Students will submit two rounds of reviews. The deadlines are

  • First round: March 16.

  • Second round: April 8.

For each round, students must review four papers:

  1. One paper, from the list below, reviewed as an individual.

  2. Three more papers, potentially reviewed together with other students, not necessarily from the list. No more than four students may be credited for a review.

Your reviews must be unique, meaning if one student or group of students submit a review of a paper, then a review of that paper by another group will not be counted. Please send me a list of papers you plan to review, by email, and I will include them in the list below. Requests are honored on a first-come-first-served basis.

Your review for each paper willl be in the form of a markdown file, using the template described below. Submit reviews on canvas either as files or a link to a repository containing your files.

Example Submission

See this page for an example review of this paper, which was generated using this file. Submit one file per paper that you review.

The files should be written in Markdown, which makes generating html easy. The first part of the sample file defines variables that are used during conversion of markdown to html:

@def review = true
@def tags = ["reviews","learning","control","analysis","optimization"]
@def reviewers = ["Hasan Poonawala","reviewer no. 2","reviewer no. 3"]
@def hasmath = true
@def class = "journal"
@def authors = "Poonawala, H.A.; Lauffer, N.;Topcu, U.;"
@def title = "Training Classifiers For Feedback Control With Safety In Mind"
@def venue = "Automatica"
@def year = "2021"

\toc

The first three lines are necessary for correct integration into the posted reviews. If you use math to convey your review, include @def hasmath = true. The remaining variables defined in lines containing @def are useful for generating an automatic summary below the review page title.

The rest of the file is the review:

\toc
### Broad area/overview
This paper deals with ...

### Notation
* $x$: ...

### Specific Problem
Instead of estimating ...

### Solution Ideas
* The paper distinguishes ...

### Comments
* The paper focuses on ...

### Recent Papers
**Note**...

Naturally, this section can vary a lot depending on the paper. You do not need to stick to the same headings or structure as the example. Providing no structure, however, is not advised.

Accessing Papers

Use the database search on the UKY Libary website to access journal databases. For example, if you type IEEE into the search you will obtain this link which takes you to the IEEE Xplore database with access through UKY, perhaps after asking for your UKY credentials.

While most links in the table below point to the official journal website, researchers often post pre-print versions that are unofficial. Websites like Google Scholar are great for finding papers with links to both official sources and pre-prints.

Pre-prints are useful for accessing papers easily, however it may be dangerous to design future research using them. There may be errors in the pre-print that were only corrected on the official peer-reviewed journal version.
